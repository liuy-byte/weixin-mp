# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 项目概述

这是一个微信公众号文章仓库（公众号：**赛博技术派** / **赛博技术派**），内容以 AI 编程工具教程为主，面向国内开发者。文章涵盖 Claude Code、OpenCode、MiniMax 模型接入、开发环境配置等实用技术指南。

本仓库不涉及代码构建、测试或 linting，主要是 Markdown 文档的编写和维护。

## 文档分类

- **Claude Code 系列**: 安装配置、命令使用、模型集成
- **OpenCode 系列**: Oh-My-OpenCode Agent 使用指南
- **开发工具**: fnm/nvm（Node.js）、jenv（JDK）版本管理
- **构建工具**: Maven Daemon（mvnd）等新一代工具介绍
- **镜像配置**: Homebrew、npm 等国内加速配置

## 公众号写作风格规范

以下为从现有文章中提炼的写作风格，编写新文章时严格遵守。

### 标题规律

- 以**数字**量化价值：「16个命令」「6大Agent」「5-10倍」
- 以**动作词**制造张力：「干翻」「告别」「让AI」
- 包含**具体收益**：「效率翻倍」「提升明显」
- 标题格式：`[产品/工具]｜[核心价值/卖点]` 或直接点明效果

示例：`赛博技术派｜用好这16个命令，Claude Code效率提升明显`

### 文章结构

```
1. 开头引用块（blockquote）—— 1-2句描述读者痛点，不超过3行
2. 效率对比表格（可选）—— 传统方式 vs 新方式，量化收益
3. 正文（汉字编号：一、二、三...）
   - 每个功能/概念独立成节
   - 核心能力：✅ 列表
   - 实战案例：> **实战案例** 引用块，含具体数据
   - 适用场景列举
4. 结语 —— 1-2句，直接点明价值，不废话
5. 免费体验（如有）—— 关注回复口令获取资源
6. 相关链接 —— 微信内长按复制格式，代码块包裹 URL
```

**严禁出现**（"AI味"特征，每次修改都要检查）：
- 模板化的"本文价值"列表（✅ 真实环境实操 / ✅ 完整命令行输出...）
- 前言里的"担心...？想...？"引导句式
- 重复说明：写了表格说明又在正文重复一遍
- 步骤解读过度细化（"1. 自动检测 Node.js，如未安装会自动安装"）
- 章节标题加"解读"、"说明"等冗余词（用"关键步骤"不用"关键步骤解读"）
- "这是**唯一需要提前准备**的内容"等强调废话
- 优化目标：比AI初稿减少约30%文字量

### 实战案例格式（固定模板）

```markdown
> **实战案例**：[具体场景描述]，[工具/方案]完成了[具体操作]，[量化结果，如"效率提升X倍"或"时间从X缩短到Y"]。
```

实战案例必须包含：具体场景 + 执行过程 + 量化效果。

### 语言风格

- **短句为主**，节奏感强，避免长难句
- **量化收益**：效率数据必须具体（「提升3.5倍」而非「大幅提升」），有实战支撑
- **口语化但专业**：「杀鸡用牛刀」「各司其职」「省心省力」
- 技术术语保留英文（JWT、OAuth2、API），但解释用中文
- 使用emoji辅助排版（✅ ❌ ⚡ 💡 🔥），不过度堆砌
- 步骤说明用简洁陈述句：「脚本自动检测并安装 Node.js 和 Git」，而非「1. 自动检测 Node.js，如未安装会自动安装」

### 格式规范

- 主要章节之间用 `---` 分隔
- 核心能力列表用 `✅` 开头
- 对比类内容优先使用表格
- 代码块内配简短中文注释
- 图片统一放 `images/` 目录，文章末尾引用

### 内容禁忌

- 不写「如有疑问欢迎留言」等无效客套
- 不写「总的来说」「综上所述」等套话过渡
- 效率数据要有实战支撑，不随意捏造
- 避免连续使用同一句式
