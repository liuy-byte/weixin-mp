## 一、用 Claude Code 的人最该兴奋

之前 200K 上下文什么体验？

一个中型项目，`src/` 目录加配置文件，轻松超 20 万 token。

你得挑文件喂。

漏了关键依赖，AI 写出来的代码一跑就报错。

因为它压根没看到完整上下文。



1M 之后呢？

50-80 万 token 的代码仓库，**整个塞进去**。

不用挑，不用选，全喂。



这意味着什么？

✅ 跨文件重构不再断片——AI 同时看到调用方和被调用方

✅ Bug 排查不用手动拼线索——整个调用链一次性摊开

✅ 新人接手直接问"这个项目架构是什么"——AI 读完全部代码给你画全景图



> **实战案例**：一个 Express + React 全栈项目，200K 窗口做重构，AI 漏掉了后端中间件依赖，生成的代码跑不起来，来回调了 3 轮。换 1M，前后端代码全部喂进去，一次完成接口重命名 + 前端适配，0 报错。

3 轮 vs 1 轮。

这就是全局视角和局部视角的差距。

## 三、搭了 RAG 的人，重新评估一下

之前 200K 装不下大量文档，只能搭 RAG。

文档切块 → 存向量数据库 → 检索相关段落 → 拼接到上下文。

一套搭下来至少一两周。



1M 之后，很多场景**可以直接跳过**。

✅ 代码库 < 80 万 token——全量喂，不用切

✅ 产品文档 + API 文档——拼起来通常不超 50 万 token

✅ 一次性分析（合同审查、竞品报告）——用完即弃

❌ 企业知识库上千份文档，远超 1M——还是得 RAG

❌ 数据实时更新（工单、日志流）——还是得 RAG

判断标准很简单：**塞得进 1M 就直接喂，塞不进再搭 RAG**。

---

## 三个坑，提前避开

**1. 不是每次都该塞满。**

写个工具函数、问个语法问题，不需要把整个项目塞进去。

1M 是上限，不是推荐用量。塞越多，响应越慢，费用越高。

**2. 中间遗忘仍然存在。**

所有大模型都有"Lost in the Middle"的毛病——开头结尾记得清，中间容易漏。

关键信息放开头或结尾，别埋在正中间。

**3. Haiku 没份。**

Haiku 4.5 还是 200K，百万上下文只有 Opus 和 Sonnet 支持。

## 行动清单

不需要改任何代码，不需要任何额外操作。

1. **Claude Code**：模型用 `claude-sonnet-4-6` 或 `claude-opus-4-6`，1M 自动生效

3. **API 调用**：`model` 参数不变，上下文自动扩到 1M
4. **搭了简单 RAG 的**：资料量 < 1M 的场景，可以砍掉 RAG 链路，省维护成本
